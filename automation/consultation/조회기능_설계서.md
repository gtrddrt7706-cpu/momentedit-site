# Moment Edit · 예약 조회 기능 구축 설계서

> **목적**: 메일 발송 한도(일반 Gmail 100통/일) 부담을 줄이기 위해, 예약번호 + 홈페이지 조회 기능을 추가한다. 진행 상황을 메일 대신 홈페이지에서 확인하게 하여 메일 사용량을 신청당 3~4통 → 1~2통으로 절감.
>
> **이 문서 하나만 새 채팅에 올리면 이어서 작업 가능. 기존 .gs/HTML 파일도 함께 업로드 권장.**

---

## 0. 배경 (왜 이걸 만드나)

- 현재 신청 1건당 메일 3~4통 발송 (고객 + contact@ + cc 미쿠·희준)
- 일반 Gmail = 하루 100통 → 신청 25~30건이면 한도 초과 ("Service invoked too many times: email")
- Workspace = 1,500통이나, 신규 도메인은 누적 $100 결제+60일 경과 전까지 100통으로 gated
- **해결**: 예약번호 발급 → 고객이 홈페이지에서 자기 예약 조회/진행 → 메일은 최소화

### 메일 절감 효과 (목표)
```
현재:  신청당 3~4통 → 하루 25건이 한계
목표:  신청당 1~2통 (고객 번호안내 + 운영자 승인알림) → 하루 50~100건 처리
```

## 2. 새 기능 설계 (조회 시스템)

### 2-1. 예약번호 (Reservation Code)
- 형식: **ME-XXXX** (예: ME-A7K2) — 사람이 읽고 입력하기 쉽게 4자리
  - 영문 대문자+숫자, 혼동 문자 제외(0/O, 1/I/L 제거) → 32^4 ≈ 100만 조합
  - 충돌 검사: 발급 시 시트에서 중복 확인, 충돌 시 재생성
- 시트에 **'예약번호' 컬럼 추가** (기존 33 → 34컬럼). 토큰과 별개:
  - 토큰(32자) = 내부 보안용 진입(메일 링크), 그대로 유지
  - 예약번호(4자) = 고객이 외우거나 입력하는 용도
- 신청 시 토큰과 함께 예약번호도 생성·저장

### 2-2. 조회 진입 (2가지 경로 병행)
```
경로 A (메일 링크 — 기존 유지): exec?page=schedule&token=xxx
   → 토큰으로 바로 내 예약. 외울 필요 없음. 메일 1통은 보냄.

경로 B (예약번호 조회 — 신규): momentedit.kr/lookup 또는 exec?page=lookup
   → 예약번호 입력 → (간단 인증) → 내 예약 페이지
```

### 2-3. 인증 방식 (예약번호만으론 약하니)
예약번호 4자리는 추측 가능성이 있어 본인 확인 필요. 옵션:
- **(권장) 예약번호 + 연락처 뒷4자리** 조합 입력 → 매칭되면 진입
- 또는 예약번호 + 이메일
- 신청 화면에서 받은 정보(연락처)로 검증 → 추가 비밀번호 불필요

### 2-4. 조회 페이지 (신규 ScreenLookup / 내 예약)
하나의 화면에서 상태별로 다르게 표시:
```
[신청접수] → "일정을 선택해 주세요" + 일정 선택 UI (ScreenB 내용 재사용)
[시간선택완료] → "디렉터 확인 중입니다" + 선택한 일정 표시
[승인완료/확정] → "예약 확정! 6/28 18:20" + 캘린더추가 + 변경/취소 버튼
[변경제안] → "디렉터가 ○○시간을 제안했습니다" + 수락/다른시간 버튼
[취소] → "취소된 예약입니다"
```
→ 이 화면이 메일 ①③④와 취소안내를 **대체**. 고객이 번호로 들어와 직접 확인.

### 2-5. 메일 최소화 후 발송 정책
```
유지(꼭 필요):
  - 신청 직후 고객 1통: 예약번호 + 조회링크 (다시 들어올 진입점)
  - 운영자 승인요청 1통: 즉시성 필요(미쿠가 폰으로)
선택(줄임/합침):
  - 확정/변경/취소 안내 → 홈페이지 조회로 대체, 메일 생략 가능(옵션 토글)
  - 운영자 cc(미쿠·희준) → 필요시만, 기본 off 검토
  - 확정 브리프 → 시트로 충분하면 생략
```
→ 신청당 2통(고객1+운영자1)으로 절감. cc까지 빼면 더 줄어듦.

## 4. 설계 결정 사항 (새 채팅에서 먼저 확정할 것)

1. **예약번호 형식**: ME-XXXX(4자) vs ME-XXXXXX(6자, 더 안전)?
2. **인증 방식**: 번호+연락처뒷4 vs 번호+이메일 vs 번호만(약함)?
3. **메일 정책**:
   - 확정/변경/취소 메일을 완전히 끌까, 아니면 짧게 1줄+조회링크로 유지?
   - 운영자 cc(미쿠·희준) 유지 vs 제거?
4. **조회 진입점**: momentedit.kr/lookup 별도 페이지 vs inquiry.html 안에 통합?
5. **기존 토큰 링크 메일**: 그대로 유지(권장) vs 번호 조회로 완전 대체?
   - 권장: 토큰 링크 + 번호 둘 다 안내 (고객 편의 + 안전)

## 5. 주의사항 (기존 시스템 보존)

- ⚠️ **기존 토큰 URL 방식은 부수지 말 것** — 잘 작동 중. 조회는 "추가"하는 방향.
- ⚠️ Apps Script 편집기 안의 inquiry.html은 **미사용**(doGet이 ScreenA/B/C만 호출). 건드리지 말 것.
- ⚠️ 시트에서 행 직접삭제 금지 → 취소는 상태="취소"로 (캘린더 자동삭제는 상태변경 트리거)
- ⚠️ 배포는 "새 버전"으로만 (새 배포 아님, exec URL 유지)
- .gs와 ScreenB는 항상 함께 배포
- 응답 스타일: 한국어·간결·플레인텍스트·구분선 ━
- 슬롯/가격/캘린더ID 등은 §1-2 값 사용
