# 구조(아키텍처) 검토 — "큰 틀에서 이게 최선인가?"

**관점:** 시스템 전체 구조가 *너의 제약(무예산 · 운영자가 시트를 직접 관리 · 전담 개발/devops 없음 · 소규모 부티크 스튜디오)* 에 최선인가, 구조적으로 현명한가.

---

## 한 줄 결론
**지금 구조는 네 제약에 매우 잘 맞는 현명한 선택이다. 갈아엎을 이유 없음.**
다만 *운영 신뢰성*을 키우는 **구조적 개선 2개(배포 자동화·청첩장 데이터 단일출처)** 가 ROI가 압도적으로 크다. 나머지는 여유 될 때.

---

## A. 구조 지도 (현재)
```
[정적 프런트 · Vercel]  index/inquiry/mypage/schedule/admin/form/live/i·i-family(템플릿)
        │  (fetch JSON, text/plain)
        ▼
[GAS 프로젝트 ②개]
  ├─ R3n9Mr  "플랫폼/상담"   consultation-booking + platform/00~85  → Customers·Consultation 시트
  └─ AKfycbwW "레터/라이브"  form-to-couple + guest-letter          → Letter System(Couples) 시트
        │   ▲ openById로 ①이 ②의 Couples에 교차 기록(청첩장 발행)
        ▼
[Google Sheets = DB]  Customers · Consultation · Couples · Signatures · Messages …
```

## B. 구조적으로 **잘 된 점** (유지할 것)
1. **빌드리스 정적 프런트** — 빠르고 싸고 튼튼. 페이지 몇 개엔 프레임워크보다 이게 맞음.
2. **Sheets = DB** — 운영자가 시트에서 직접 보고 고침. 소규모 스튜디오엔 *최강의 장점*(무료·투명·백오피스 공짜).
3. **하이브리드 단계 모델** — 선형 `현재단계` + 병행 상태(시착·계약·입금·제작·결과물). 잔금도 이 패턴에 자연히 들어맞음(= 설계 일관성 증거).
4. **안전장치** — 단일 전이점(setCustomerStage)·데이터 보존·강제변경+사유+이력·멱등·LockService. 견고함.
5. **관심사 분리** — 운영(플랫폼) vs 청첩장 렌더(레터). 개념적으론 옳음.

→ 이 5개는 *구조적으로 현명*. 바꾸지 말 것.

## C. 구조적 **개선점** (ROI 높은 순)

### ★ C1. GAS 배포 자동화 (clasp / CI) — **최우선**
- **문제:** `.gs` 수동 복붙 + 재배포 → 누락·드리프트·실수. 우리가 직접 겪음(5개 파일 업로드, LETTER_SYSTEM 미설정, "전부 배포했나?"). **repo의 .gs ↔ 실제 배포 코드가 조용히 갈라질 수 있음**(가장 위험).
- **해결:** **clasp**(구글 공식 CLI)로 `repo → GAS` push. 또는 GitHub Action으로 자동화. "파일 업로드" 의식 자체가 사라지고 **항상 일치**.
- **내가 할 수 있는 것:** `.clasp.json`·`appsscript.json` 스캐폴드 + 셋업 가이드. **네가 1회:** `clasp login`(구글 인증)·`clasp push`.
- ROI **매우 높음**(반복 고통 제거·드리프트 0) / 노력 중(1회).

### ★ C2. 청첩장 데이터 **단일 출처** — **높음**
- **문제 ①(중복):** 이름·예식일이 **Customers(제작임시저장)** 와 **Couples(41열)** 에 이중 저장 → 한쪽만 고치면 **불일치**(예: 이름 오타 수정이 청첩장에 반영 안 됨).
- **문제 ②(이중 코드):** Couples 행을 만드는 경로가 **둘** — 옛 구글폼(form-to-couple) + 마이페이지 위저드(publishInvitation). `makeEventId` 등이 **"복제"**(코드 주석에도 명시)라 둘이 갈라지면 버그(과거 날짜검증 버그가 이 계열).
- **해결:** (a) "Couples 행 빌드" 로직을 **한 함수로 통합**(폼·위저드 공용 호출), (b) 필드별 **정본 1곳** 지정 + 단방향 동기화. 장기엔 **폼 경로 폐기 → 마이페이지 단일 깔때기**.
- ROI 높음(오타·불일치 버그 예방) / 노력 중.

### C3. 프런트 **공통화**(토큰·코어 유틸) — 중
- **문제:** `:root` 디자인 토큰과 `api()`·`escapeHtml()`·토큰 헬퍼가 mypage·admin·schedule·템플릿에 **각각 복제** → 드리프트(실제로 mypage엔 `--gold-bg`가 없어 폴백으로 때웠음).
- **해결:** `/shared/tokens.css` · `/shared/core.js`로 추출해 전 페이지 `link`/`script`. **빌드리스 유지하면서 단일 출처.**
- ROI 중(테마·로직 일관·수정 1곳) / 노력 소~중.

### C4. 조회·트리거용 필드만 **컬럼 승격** — 중
- **문제:** `제작임시저장` JSON 블롭은 시트에서 **필터/트리거 불가**(잔금 PLAN이 `예식일` 컬럼을 승격하는 이유).
- **해결:** 운영/트리거/필터에 쓰는 **소수 필드만 컬럼화**(예식일·트랙상태), 나머진 JSON 유지. 유연성 + 조회성 균형.
- ROI 중 / 노력 소. (잔금 PLAN에 일부 포함)

### C5. 경량 CI 게이트 — 중
- `node --check`(인라인 JS 문법) + (clasp 도입 시)push를 **GitHub Action**으로 → 깨진 코드 배포 예방. 지금은 내가 수동으로 문법검사 중.

## D. **안 바꾸는 게 현명한 것** (과설계 경고)
- **React/프레임워크 도입 ❌** — 빌드·복잡도만 늘고 이득 적음(정적 페이지 소수).
- **진짜 DB(Postgres 등)·전용 서버 ❌** — *운영자 직접 편집·무료*라는 최대 장점을 잃음. 현재 규모엔 Sheets로 충분.
- **마이크로서비스/큐 ❌** — 명백한 과설계.
- 즉, "더 그럴듯한 스택"으로 가는 건 **이 비즈니스엔 오히려 후퇴.**

## E. 규모 한계 (언제 재검토?)
- GAS+Sheets는 대략 **행 수천~1만 / 일일 메일·트리거 쿼터** 내에서 쾌적.
- **재검토 신호:** 연 수천 건↑, getMyState가 캐시·TextFinder로도 느림, 메일 쿼터 한계, 동시 편집 충돌 빈발.
- **지금은 그 한계에서 한참 멀다** — 선제 이주 불필요.

## F. 리스크 한눈
| 리스크 | 현재 | 완화 |
|---|---|---|
| 배포 드리프트(repo↔GAS) | **중** | C1(clasp) |
| 청첩장 데이터 불일치 | **중** | C2(단일출처) |
| GAS 단일 장애점(다운/쿼터) | 저~중 | 읽기 SWR 캐시(mypage 적용·schedule로 확대), 쿼터 모니터 |
| 프런트 토큰/유틸 드리프트 | 저 | C3 |
| 보안(토큰·XSS) | 저 | escapeHtml 일관·HMAC 재설정(이미 양호) |

## 결론 & 추천 순서
큰 틀은 **최선에 가깝다.** 갈아엎지 말고 **두 개만** 잡으면 구조 신뢰성이 확 오른다:
1. **C1 — 배포 자동화(clasp)**: 반복 고통·드리프트 제거. *체감 1위.*
2. **C2 — 청첩장 데이터 단일 출처**: 데이터 불일치 버그 뿌리 뽑기.

그 다음 여유 되면 C3(프런트 공통화) → C4(컬럼 승격) → C5(CI).

> 원하면 **C1부터** 착수할게요 — `.clasp.json`·`appsscript.json` 스캐폴드 + "한 번만 하면 되는" 셋업 가이드(클론→`clasp login`→`clasp push`)를 만들어 드리고, 그 뒤론 `.gs` 업로드가 사라져요. (C2는 설계가 좀 더 필요해서 PLAN 문서로 이어서.)
