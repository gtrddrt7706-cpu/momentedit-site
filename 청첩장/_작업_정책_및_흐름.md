## 2. 청첩장 종류

| 종류 | 용도 |
|---|---|
| **live** (라이브 청첩장) | 일반 하객용. 모바일 페이지로 발송. 모바일 최적화 |

## 4. 부모 이름 표시 정책 (실무자 토글 항목)

청첩장에 부모 이름이 들어가는 위치는 **2곳**:
- **위쪽** — 인사글 끝부분의 자녀 소개 부분
- **아래쪽** — 마음 전하실 곳 (계좌 안내)

### 토글 항목 (2개)
| 시트 컬럼 | 값 | 결과 |
|---|---|---|
| `greetingShowParents` | YES | 인사글에 부모 이름 표시 |
| | NO | 인사글에 본인 이름만 표시 |
| `envelopeShowParents` | YES | 계좌 안내에 부모 계좌까지 표시 |
| | NO | 계좌 안내에 본인 계좌만 표시 |

→ 2 × 2 = **4가지 경우**:
| # | 위쪽 | 아래쪽 | 실무 빈도 |
|---|---|---|---|
| 1 | ✅ 부모 이름 | ✅ 부모 계좌 | 가장 흔함 (기본) |
| 2 | ✅ 부모 이름 | ❌ 본인 계좌만 | 흔함 (부모 자금 분리) |
| 3 | ❌ 본인만 | ✅ 부모 계좌 | 드묾 (모던 인사글 + 부모 계좌) |
| 4 | ❌ 본인만 | ❌ 본인만 | 흔함 (미니멀 톤) |

## 5. 인사글 본문 정책

### 입력 가능, 비워두면 기본값
| 시트 컬럼 | 동작 |
|---|---|
| `invitationText` (비어 있음) | 디자인별 기본 인사글 자동 사용 |
| `invitationText` (직접 입력) | 입력 텍스트로 치환 |

### 디자인별 기본 인사글 톤
| 디자인 | 기본 인사글 톤 |
|---|---|
| 01 Classic | 짧고 정중 |
| 02 Editorial | 모던 |
| 03 Letterpress | 격식 |
| 04 Vermilion | (인사글 영역 없음 — 표지 카피로 처리) |
| 05 Botanical | 부드러움 |
| 06 Hangeul | 전통적 한국어 |
| 07 Architect | 격식 모던 |
| 08 Noir | (인사글 영역 없음 — BIO 카드로 처리) |

### 04, 08 특이 사항
- 04와 08은 디자인 정체성상 인사글 영역이 없음
- 이 두 디자인 선택 시 `invitationText` 입력해도 표시할 곳 없음
- 시트에서는 입력 가능하되 시스템이 무시 (실무자는 인사글 영역 없는 디자인임을 사전 인지)

## 6. 실무자 입력 항목 정리

### 디자인 선택
- `designId` (01~08 중 택1)
- 선택 시 라벨 패턴(A/B/C/D), 외각 처리, 기본 인사글 톤 모두 자동 결정

### 부모 이름 표시
- `greetingShowParents` (YES/NO)
- `envelopeShowParents` (YES/NO)

### 인사글 본문
- `invitationText` (빈칸 또는 직접 입력)

### 청첩장 종류
- `cardType` (live / family)

### 그 외 (필수 정보)
- 신랑/신부 이름, 결혼일, 식장 정보, 본인 계좌, 부모 이름·계좌 (`envelopeShowParents=YES`인 경우)

## 7. 빌드 시스템

### 파일 구조
```
/home/claude/work/
├── live-01.master ~ live-08.master       # 8개 라이브 마스터
├── family-01.master ~ family-08.master   # 8개 가족 마스터

└── build_all_16.py                       # 시뮬레이션 빌드 스크립트
```

### 시뮬레이션 빌드
- `build_all_16.py` 실행 → 16개 HTML 결과를 `/mnt/user-data/outputs/_review_*.html`에 생성

- 휴대폰/PC로 확인용

### 실제 빌드 (Phase C에서 구현 예정)
- Google Sheets에 실무자가 입력
- Apps Script가 행 데이터 → 마스터 HTML로 자동 치환
- 결과 HTML 파일 생성

## 8. 변경 작업 시 안전 원칙

### 작업 전
1. `finalized/` 디렉토리에 백업 사본이 있는지 확인
2. 현재 빌드 결과(`/mnt/user-data/outputs/_review_*.html`)를 별도 백업

### 작업 중
3. 한 번에 1개 디자인 또는 1개 시스템 영역만 수정
4. 매 변경 후 재빌드 + 휴대폰 캡처로 시각 확인

### 작업 후
5. 빌드 결과 diff 비교 (백업 vs 작업 후)
6. 의도하지 않은 시각 변경 감지 시 즉시 롤백
7. 확정되면 `finalized/`에 동기화

## 9. 앞으로 작업 계획 (Phase C/D/E)

### Phase C — Apps Script 작성 (최우선, 가장 큼)
**목표**: 시트에 실무자가 입력 → 자동으로 HTML 청첩장 생성

세부 작업:
1. **시트 컬럼 확정** — 위 정책에 맞게 컬럼 정의 (디자인 선택, 부모 토글 2개, 인사글, 신랑/신부 정보, 식장, 계좌 등)
2. **`라이브청첩장빌드.gs`** 신규 작성 — 8개 디자인 모두 지원
3. **`MomentEdit_FamilyCard_Build_v6.gs`** 갱신 — 새 정책 반영
4. **OPTIONAL 처리 로직** — `greetingShowParents`, `envelopeShowParents`에 따라 6개 OPTIONAL 주석 자동 처리
5. **인사글 처리 로직** — 입력값 있으면 치환, 없으면 마스터 기본 유지
6. **변수 치환** — 16개 마스터의 77개 변수(`{{GROOM_NAME}}` 등)를 시트 데이터로 치환

### Phase D — 테스트
1. 4가지 부모 표시 케이스 각각 검증
2. 8개 디자인 모두에서 빌드 가능한지 확인
3. live ↔ family 양쪽 다 정상 빌드
4. 모바일·태블릿·PC 렌더링 확인

### Phase E — 시스템 문서 갱신
1. `01_AI_시스템_마스터.html` — 이번 정책 반영
2. `02_실무자_작업_매뉴얼.html` — 실무자 작업 흐름 갱신
