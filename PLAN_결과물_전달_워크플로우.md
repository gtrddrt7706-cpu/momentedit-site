# PLAN · 결과물 전달 워크플로우 (원본 → 고객 선택 → 보정 → 전달)

> 작성: 결과물(05) 단계를 "원본 전달 → 고객이 고르기 → 관리자 확인 → 보정 후 전달 → (추가 보정·결제)"
> 한 흐름으로 설계. **마이페이지(고객) · 관리자 페이지 양쪽을 기획 단계부터** 같이 본다.
> 코드 근거: `mypage.html renderResult()` / `admin.html cardResult()` / `automation/platform/80_production.gs buildResultState()` / `00_platform-config.gs`.

---

## 0. 한 줄 요약

지금은 결과물이 **"링크만 등록 → 전달 완료"** 2단계다. 여기에 **고객이 직접 고르는 ②번**과
**관리자가 그 선택을 보고 보정해서 올리는 흐름**, **추가 보정·결제 ③번**을 끼워 넣어
*원본 → 선택 → 보정 → 전달*이 한 줄로 이어지게 만든다.

## 1. 현재 상태 (코드 기준)

### 데이터 (`00_platform-config.gs`)
- `결과물상태` enum: **대기 · 업로드 · 전달완료** (3단계, 거칠다)
- 링크 컬럼: **원본링크 · 보정본폴더 · 영상링크**
- 단계(`현재단계`): 예식완료 / 촬영완료 → (최종) 결과물전달

### 마이페이지 `renderResult(res)`
- `res.delivered === false` (예식완료/촬영완료): "결과물 준비 중" 안내 + 예상 기간(`DELIVERY_ETA`, 현재 `?`)
- `res.delivered === true` (결과물전달): 5스텝 표시
  - ① 원본 사진 — `res.원본` 링크
  - ② 보정 사진 고르기 — **"곧 열려요" 자리(미구현)**
  - ③ 추가 보정·결제 — **"곧 열려요" 자리(미구현)**
  - ④ 보정본 — `res.보정본` 링크
  - ⑤ 본식 영상 — `res.영상` 링크 (스냅은 없음)

### 관리자 `cardResult(d)`
- `결과물상태` + 원본/보정본/영상 유무(✓/—) + 링크
- 액션: **결과물 링크 등록**(`adminSetResultLinks`: 원본/보정본/영상 URL) · **결과물 전달**(`adminMarkDelivered`, 원본 필수) · 예식/촬영완료 처리(`adminMarkEventDone`)

### 빈틈 (= 이번에 채울 것)
| 원하는 흐름 | 지금 | 채울 것 |
|---|---|---|
| 원본 전달 | ✅ 원본링크 등록 | 그대로 활용 |
| 고객이 고르기 ② | ❌ "곧 열려요" | **선택 UI + 제출** |
| 관리자가 선택 확인 | ❌ 없음 | **선택 목록 보기** |
| 고른 컷 보정 후 전달 ④ | △ 보정본폴더 링크만 | 선택과 연결 + 상태 분리 |
| 추가 보정·결제 ③ | ❌ "곧 열려요" | **신청 → 견적 → 결제** |

## 2. 전체 흐름

```
[관리자]              [고객·마이페이지]            [관리자]
예식/촬영 완료 처리
      │
      ▼
원본 업로드(Drive)  ──▶  ① 원본 보기
원본링크 등록            ② 마음에 드는 컷 고르기 ──▶  선택 목록 확인
(상태: 원본전달)           (상태: 선택완료)              "○○님이 N장 선택"
                                                          │
                                                          ▼
                                                    고른 컷만 보정
                          ④ 보정본 받기  ◀──  보정본 업로드 + (영상)
                          (상태: 전달완료)        결과물 전달 처리
                                                  → 아카이브
                          ③ 더 보정받고 싶으면
                             추가 보정 신청  ──▶  견적 → 결제확인 → 보정
```

핵심 원칙
- **사진 파일 자체는 서버에 두지 않는다.** 원본·보정본은 계속 Google Drive 폴더. 우리 시트엔 *링크·선택목록·상태*만.
- **단계(`현재단계`)는 끝까지 예식완료/촬영완료**로 두고, 내부 진행은 `결과물상태`로 표현 →
  마지막 보정본 전달 때 한 번에 `결과물전달`로 넘기고 아카이브(지금의 안전한 동작 유지).

## 3. 상태 모델 — `결과물상태` 확장

| 상태 | 의미 | 트리거 | 마이페이지 | 관리자 |
|---|---|---|---|---|
| `대기` | 원본 아직 없음 | 예식/촬영완료 직후 | "준비 중" 안내(기간) | [원본 등록] |
| `원본전달` | 원본 올라옴, 고를 수 있음 | 관리자 원본링크 등록 | ① 보기 + **② 고르기(활성)** | "고객 선택 대기" |
| `선택완료` | 고객이 컷 제출 | 고객 선택 제출 | ② ✓ N장 + ④ "보정 중" | **선택 목록 확인** + [보정본 등록] |
| `보정중`(선택) | 관리자 보정 작업 중 | 관리자 표시(선택) | ④ "보정 중" | 진행 표시 |
| `전달완료` | 보정본(+영상) 전달 | 관리자 전달 처리 | ④⑤ 받기 + **③ 추가 보정** | ✓ 전달 완료 |

> 기존 `업로드` 상태는 `원본전달`로 대체(의미 명확화), `전달완료`는 유지. 마이그레이션에서 `업로드`→`원본전달` 치환.

## 4. 데이터 모델 — 추가 컬럼

`Customers` 시트에 추가(멱등 마이그레이션 `addResultSelectionColumns()`로 없을 때만 생성):

| 컬럼 | 타입 | 용도 |
|---|---|---|
| `선택사진` | text | 고객이 고른 컷. **1차: 번호/파일명 CSV** / 2차: 파일ID CSV |
| `선택수` | number | 고른 장수(관리자 한눈에) |
| `선택확정일시` | datetime | 제출 시각 |
| `보정완료일시` | datetime | (선택) 보정본 전달 시각 |
| `추가보정상태` | enum | 대기 / 신청 / 견적 / 결제대기 / 완료 |
| `추가보정수량` | number | 추가로 보정 신청한 장수 |
| `추가보정금액` | number | 견적 금액 |
| `원본폴더ID` | text | (2차·B안) Drive 폴더ID — 썸네일 갤러리용 |

기존 `원본링크`(원본 폴더 공유링크) · `보정본폴더`(보정본 폴더 공유링크) · `영상링크`는 그대로.

## 5. 사진 선택 메커니즘 — 3안 비교 ★결정 필요

원본 사진은 Drive에 있는데, 고객이 "그중 고르기"를 어떻게 하느냐가 이 시스템의 갈림길.

### A안 — 번호/파일명 입력 (Drive 링크 기반) · **빠른 가동**
- 관리자: 원본을 Drive 폴더에 올리고 **공유 링크만** 등록(지금과 동일).
- 고객: 마이페이지 [원본 보기]로 Drive에서 보고, **고르기 칸에 원하는 번호/파일명 입력**(예: `012, 045, 088`) → 제출.
- 관리자: `선택사진` 목록 그대로 확인 → 해당 컷 보정.
- ➕ **드라이브 API·썸네일·서버 부하 0, 즉시 구현**, 기존 원본링크 그대로 사용.
- ➖ 번호 수기 입력(오타 가능), 마이페이지 안에 썸네일은 없음(드라이브에서 봄).

### B안 — 자체 썸네일 갤러리 (Drive API) · **프리미엄**
- 관리자: 원본 폴더 **폴더ID** 등록.
- GAS `DriveApp`이 폴더 내 파일을 열거 → 썸네일/미리보기 URL 목록 반환.
- 고객: 마이페이지에서 **썸네일 그리드 + 하트/체크로 탭 선택** → 파일ID 목록 제출.
- 관리자: 고른 컷을 **썸네일로** 확인(어떤 사진인지 바로 보임).
- ➕ 청첩장급 셀렉 경험, 오타 없음, 관리자도 썸네일 확인.
- ➖ Drive 권한/공유 설계, 수백 장 썸네일 로딩(페이징 필요), 구현량·GAS 호출 비용 큼.

### C안 — 외부 셀렉 서비스(픽플러·포토셀렉 등)
- 외부에 업로드/셀렉, 결과 수량만 우리 시트에 기록.
- ➕ 구현 최소, 검증된 UX. ➖ 외부 의존·비용, 마이페이지 이탈, "우리 시스템" 아님.

### ▶ 추천: **A안으로 먼저 가동 → B안으로 진화**
`선택사진/선택수/선택확정일시` 데이터는 A·B 공통이라 **A안이 헛수고가 안 됨**. 빠르게 흐름을 돌리고,
반응 보고 B안(썸네일 갤러리)으로 업그레이드. 품질 지향이면 B안이 목표지만, 사진 수백 장 성능 설계가 필요해 2차가 안전.

## 6. 마이페이지(고객) 화면 — 상태별

`renderResult()`의 분기 기준을 `res.delivered` → **`res.status`(결과물상태)** 로 바꾼다.

- **대기**: 지금의 "결과물 준비 중" 안내(기간 `DELIVERY_ETA`) 그대로.
- **원본전달**:
  - ① 원본 사진 — [원본 보기 ↗] (Drive)
  - ② **마음에 드는 컷 고르기 (활성)**
    - A안: 안내문("드라이브에서 보고 파일 번호를 적어 주세요") + textarea + [고른 컷 제출] → 확인 모달
    - B안: 썸네일 그리드, 탭하면 하트 토글, 상단에 "N장 선택됨", [제출]
  - ③④⑤ — "곧 열려요" 자리
- **선택완료/보정중**:
  - ① 원본 / ② **✓ N장 선택 완료**(다시 고르기 가능?—§11) / ④ "한 장씩 곱게 보정 중이에요" 대기 안내
- **전달완료**:
  - ① 원본 / ② ✓ / ③ **추가 보정·결제(활성)** / ④ 보정본 받기 / ⑤ 영상

문구 톤은 기존 그대로(따뜻하게). ②는 "고르신 컷만 보정해 드려요. 천천히 고르셔도 돼요." 같은 안심 카피.

## 7. 관리자 페이지 화면 — 상태별 (`cardResult` 확장)

현재 카드(상태 + 링크 유무 + [링크 등록]/[전달])에 **선택 확인 블록**과 **단계 버튼**을 추가.

- **대기**: [원본 등록] (= 기존 setResult의 원본 칸; 등록 시 `결과물상태=원본전달`)
- **원본전달**: "🕓 고객 선택 대기 중" 표시. (재촉 알림 보내기 — 선택)
- **선택완료**:
  - **▣ 고객이 고른 컷** — A안: `선택사진` 텍스트 그대로(번호 N장) / B안: 선택 썸네일 그리드
  - "N장 선택 · ○월 ○일 제출"
  - [보정본 폴더 등록] → 등록 후 [결과물 전달]
- **전달완료**: ✓ 전달 완료 + (③ 추가 보정 신청 있으면) **"추가 보정 N장 신청 — 견적/확인"** 블록
- 공통: 원본/보정본/영상 링크 chip 유지.

> 관리자 "결과물 링크 등록" 모달을 **원본 등록**과 **보정본·영상 등록** 2가지 맥락으로 분리하면 흐름이 또렷해진다(같은 `adminSetResultLinks` 재사용, 모달 카피만 분기).

## 8. API 액션 명세

### 마이페이지(고객) — `consultation-booking.gs` doPost 라우터 + 핸들러
| action | 입력 | 처리 |
|---|---|---|
| `submitResultSelection` | token, picks(문자열/배열) | `선택사진/선택수/선택확정일시` 기록, `결과물상태=선택완료`. 스튜디오 알림. |
| `requestExtraRetouch` | token, qty, note | `추가보정상태=신청`, `추가보정수량=qty`. 스튜디오 알림. |
| (B안) `getResultGallery` | token | `원본폴더ID`로 DriveApp 열거 → [{id, name, thumb}] |

### 관리자 — `admin.gs` adminCall 레지스트리
| action | 입력 | 처리 |
|---|---|---|
| `adminSetResultLinks` | code, {원본,보정본,영상} | (기존) 원본만 등록 시 `원본전달`, 보정본 등록 시 보정 단계 |
| `adminGetSelection` | code | `선택사진/선택수/선택확정일시` 반환(B안: 썸네일) — 또는 기존 adminDetail mirror에 포함 |
| `adminConfirmExtra` | code, 금액 | `추가보정상태=견적/완료`, `추가보정금액` 기록 |
| `adminMarkDelivered` | code | (기존) 보정본 전달 + 아카이브 |

`buildResultState(r)`에 `status(결과물상태)`, `선택수`, `선택사진`, `추가보정*`를 추가 반환.
`adminDetail`의 `mirror.result`(또는 raw)에도 선택 정보를 실어 관리자 카드가 바로 읽게.

## 9. ③ 추가 보정·결제 설계

- **개념**: 계약 포함 보정 컷 수(예: N장)를 넘겨 더 받고 싶을 때.
- **흐름**: 고객 *추가 수량 신청* → 관리자 *단가×수량 견적* → 고객 *입금* → 관리자 *확인* → 보정.
- **결제 재사용**: **잔금 흐름과 동일 패턴**(계좌 입금 → 고객 "입금했어요" 신호 → 관리자 확인). 새 결제수단 안 만든다.
- **데이터**: `추가보정상태(대기/신청/견적/결제대기/완료) · 추가보정수량 · 추가보정금액`.
- **결정 필요**: 계약 *포함 컷 수*, *추가 단가*. (정해지면 자동 견적, 미정이면 "관리자가 안내" 수동.)

## 10. 단계적 구현 로드맵

| Phase | 내용 | 백엔드(GAS) | 프론트 | 비고 |
|---|---|---|---|---|
| **1 · 가동** | A안 선택 + 관리자 확인 + 상태 세분화 | 컬럼 추가, `submitResultSelection`, `buildResultState` 확장, `결과물상태` enum 확장 | mypage ② 입력폼, admin 선택 확인 블록 | 데이터는 2차에서 재사용 |
| **2 · 프리미엄** | B안 썸네일 갤러리 | `getResultGallery`(DriveApp), 폴더ID | mypage 썸네일 셀렉, admin 썸네일 확인 | 페이징·권한 설계 |
| **3 · 추가결제** | ③ 추가 보정·결제 | `requestExtraRetouch`, `adminConfirmExtra` | mypage ③ 신청·입금, admin 견적·확인 | 잔금 패턴 재사용 |

각 Phase는 **null-safe**로 배포 → 프론트가 먼저 떠도(데이터 없으면 해당 블록 숨김) 깨지지 않음.

## 11. 엣지 케이스 & 안전장치

- **선택 변경**: 전달 전까지 다시 고르기 허용(덮어쓰기 + `선택확정일시` 갱신). 보정 시작 후엔 잠그고 "변경은 문의" 안내.
- **부분 전달**: 보정본만 먼저, 영상은 나중 — 지금처럼 "하나씩 올라와요" 유지(전달완료는 보정본 기준, 영상은 추가 등록).
- **원본 미등록 전달 차단**: 기존 가드(원본 없으면 전달 불가) 유지.
- **선택 0장 제출 방지**: 최소 1장. 너무 많을 때 안내(포함 컷 수 초과 → ③ 유도).
- **만료/아카이브**: 전달완료 + 아카이브 동작은 그대로.
- **대용량(수백 장)**: A안은 영향 없음. B안은 썸네일 페이징·지연로딩 필수.
- **드라이브 권한(B안)**: 고객이 전체 폴더를 못 보게 하면서 썸네일만 노출할지, 폴더 공유로 갈지 결정 필요.

## 12. 결정 — 확정 & 구현됨 ✅

1. **선택 방식 → A안(번호 입력)으로 1차 가동 → B안(썸네일 갤러리) 2차.** (사용자: "추천대로")
2. **③ 추가 보정 → 포함 10컷 / 추가 컷당 ₩20,000.** momentedit.kr 가격표에서 확인해 그대로 대입
   (`웨딩스냅 원본 전체 + 보정 10장`, `추가 보정 ₩20,000/컷` — 시그니처·스냅 공통). 자동 견적(수량×20,000).
   → 코드 단일 출처: `80_production.gs RESULT = { 포함보정컷:10, 추가보정단가:20000 }`.
3. **전달 매체 → Google Drive 폴더 링크 유지(추천).**
   - 이유 ①: A안과 바로 맞물림(고객이 드라이브에서 파일명·번호 보고 선택).
   - 이유 ②: **B안으로 갈 때 같은 폴더를 Drive API로 그대로 읽어** 썸네일 갤러리로 진화 → 전환이 매끄러움(이사 X).
   - 이유 ③: 외부 셀렉 서비스(픽플러 등)는 마이페이지 밖으로 이탈 + 비용 → "우리 시스템·프리미엄" 방향과 안 맞음.
   - 운영 팁: 원본 파일명을 **번호순(001, 002 …)** 으로 정리하면 A안 번호 입력이 깔끔(오타↓).

### ▶ Phase 1(A안) 구현 완료
- GAS: `결과물상태` 세분화(대기/원본전달/선택완료/보정중/전달완료) + 컬럼(선택사진·선택수·선택확정일시·추가보정상태/수량/금액)
  + `handleSubmitResultSelection`·`handleRequestExtraRetouch`·`handleExtraRetouchSignal`·`adminConfirmExtra`
  + `buildResultState` 확장 + 마이그레이션 `addResultSelectionColumns()`.
- 마이페이지: 결과물 카드 = 원본①→선택②(번호 입력·수정 가능)→추가 보정③(신청·입금)→보정본④→영상⑤, 상태별 분기.
- 관리자: 결과물 카드에 **고른 컷 목록**·**추가 보정 신청** 표시 + [추가 보정 입금 확인].
- 무회귀: 백엔드 배포 전엔 기존 카드 그대로(폴백). 배포 후 자동으로 새 흐름.
