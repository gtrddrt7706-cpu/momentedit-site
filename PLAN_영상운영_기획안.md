# PLAN · 본식 영상 등록·운영 기획안 (디지털 참석)

작성 2026-07-01 · 대상: `live.html` 영상 송출 + 온라인 청첩장 Enter 흐름
목표: **하객이 예식 시각에 링크만 열면 영상이 보이고, 운영자(1인)는 당일 실수 여지가 없는** 표준 운영 체계 확정.

---

## 0. 현재 시스템 요약 (파악 완료)

```
청첩장 i/cover-0X.html?e=ID → [Enter] → /live.html?e=ID
  → getCouple API (guest-letter-webhook.gs · 별도 GAS · view=live)
  → Couples 시트 vimeoId·vimeoHash
  → 값 있으면 Vimeo iframe (autoplay·muted·1080p) / 없으면 "예식 시각에 시작됩니다" 플레이스홀더
```

- 부부 등록: 구글폼 → `form-to-couple.gs`가 Couples 행 생성 + 청첩장·live URL 부부 메일 발송. **vimeoId는 폼에 없음 → 운영자가 나중에 직접 입력하는 유일한 수동 필드.**
- 캐시 2겹: 서버 CacheService 600초(`?fresh=1`로 우회+재적재) · 하객 브라우저 localStorage(재방문 즉시 렌더용).
- 라이브 종료 후에도 같은 링크가 그대로 다시보기가 되는 구조(장점).

---

## 1. 결정할 것 3가지

### 결정 ① 송출 방식

| | A. 진짜 라이브 | B. 준라이브(녹화 후 빠른 업로드) | C. 다시보기 전용 |
|---|---|---|---|
| 하객 경험 | 예식 시각 실시간 참석 ★기획 의도 | 당일 저녁~다음날 시청 | 원하는 때 시청 |
| 당일 부담 | 높음(송출 장비·네트워크) | 낮음(촬영만) | 없음 |
| 실패 리스크 | 네트워크 끊김 | 거의 없음 | 없음 |
| Vimeo 요금 | 라이브 지원 상위 플랜 필요 | 하위 플랜으로 충분 | 하위 플랜 |
| 비고 | 종료 후 자동으로 다시보기 전환 | placeholder 문구와 어긋남(수정 필요) | 〃 |

- 페이지 카피("본식 라이브"·"예식 시각에 시작됩니다"·"Moment Edit Live · 1080p")가 전부 **A 전제**로 쓰여 있음. B/C로 가면 카피도 함께 수정해야 함.
- **권장: A(라이브)를 기본 상품으로, 네트워크 불가 예식장만 B로 다운그레이드**(사전답사 때 판정).

### 결정 ② 플랫폼

| | Vimeo (현행 코드 그대로) | YouTube 비공개(unlisted) |
|---|---|---|
| 비용 | 라이브는 상위 플랜(대략 월 $60~75·연간결제, 결제 전 최신가 확인) | 무료 |
| 브랜딩 | 로고·추천영상 없음, 임베드 도메인 제한 가능 ★ | 유튜브 UI·관련영상 노출, 프라이빗 결이 깨짐 |
| 코드 | 수정 0 (vimeoId·vimeoHash 그대로) | live.html에 youtubeId 분기 추가 필요 |
| 비공개성 | unlisted+hash+도메인 임베드 제한 = 강함 | unlisted 링크 아는 사람 모두, 유튜브 검색엔 안 뜸 |
| 라이브→다시보기 | 같은 ID 유지(자동) | 같은 ID 유지(자동) |

- **권장: Vimeo 유지.** "가까운 분들만 비공개" 약속과 프리미엄 톤이 상품 핵심이라 유튜브 UI 노출은 감점이 큼. 단, 초기 예식 건수가 적어 고정비가 부담이면: 첫 1~2건은 B(준라이브·하위 플랜)로 시작하고 라이브 계약이 잡히는 달만 상위 플랜 결제(월 단위 업/다운) 전략도 가능.

### 결정 ③ 등록 시점 — ★이 기획안의 핵심

| | 당일 입력(현행 안내) | **사전 등록(D-3) ★권장** |
|---|---|---|
| 방법 | 예식 직전 시트 N·O열에 입력 | Vimeo에서 라이브 이벤트를 미리 만들면 영상 ID·hash가 그 자리에서 확정 → D-3에 시트 입력 |
| 캐시 영향 | 서버 600초 + 하객 재로딩 필요 → **당일 지연·혼선** | 며칠 전 전파 완료 → 캐시 문제 소멸 |
| 당일 할 일 | 시트 입력+확인 (가장 바쁜 시간에) | **0** (송출 시작만) |
| 하객 화면 | 입력 전엔 placeholder | 입력 후엔 Vimeo 대기 화면(스트림 시작 전) → 시작하면 자동 재생 |

- Vimeo 라이브 이벤트는 **만드는 순간 영상 ID가 확정**되고, 방송 전엔 대기 화면, 방송 시작 시 같은 임베드에서 자동 재생, 종료 후 같은 ID로 아카이브(다시보기)까지 이어짐. → **vimeoId 하나로 '대기 → 라이브 → 다시보기' 전 생애주기가 자동.**
- 유일한 트레이드오프: D-3부터 하객이 열면 자사 placeholder 대신 Vimeo 대기 화면이 보임. 신경 쓰이면 §3-P3의 "예식 90분 전까지 placeholder 유지" 게이트를 넣으면 해결.

## 2. 표준 운영 절차 (SOP · 권장안 기준)

**계약 시** 폼 제출 → Couples 행·청첩장 URL·부부 메일 자동(기존 그대로). 예식장 네트워크 사전답사 항목에 "업로드 5Mbps 이상 확보 가능?" 추가 → 불가면 준라이브로 전환 합의.

**D-7** Vimeo에서 해당 부부 라이브 이벤트 생성(제목 `YYMMDD_신랑신부`). 프라이버시: 링크 소지자만+임베드 momentedit.kr 한정. 스트림 키 메모.

**D-3** Couples 시트 N열 vimeoId · O열 vimeoHash 입력 → `live.html?e=ID&fresh=1` 한 번 열어 확인(이 호출이 서버 캐시도 새 값으로 재적재함 → 이후 모든 하객에게 즉시 전파).

**D-1 리허설(15분)** 장비로 스트림 키에 테스트 송출 → 본인 폰에서 `live.html?e=ID` 열어 화면·소리·가로화면비 확인 → 송출 중단(대기 화면 복귀 확인). 배터리·저장공간·삼각대 체크.

**당일**
- H-90분: 장비 설치(폰+짐벌 or 캠+노트북 OBS), 네트워크 확인(가능하면 유선/전용 테더링, 행사장 공용 와이파이 금지)
- H-40분: 송출 시작 상태로 대기(대기 화면) → 하객 입장 링크 정상 확인
- H-0: 본식 송출(~30분, 크레딧바 표기와 일치)
- 종료: 송출 중단 → 같은 링크가 자동으로 다시보기 전환 → 부부에게 "다시보기 링크는 동일합니다" 안내(알림톡/메일 문구 추후)

**D+30(정책 결정 필요)** 다시보기 유지 기간·원본 파일 납품 여부를 상품 약관에 명시(예: 30일 유지 후 비공개 전환, 원본은 Drive 납품).

**장비 최소 구성(1인)**: ① 간편: 폰(Larix Broadcaster·RTMP)+짐벌+무선마이크 ② 화질우선: 미러리스+HDMI 캡처보드+노트북 OBS. 둘 다 스트림 키만 넣으면 됨.

## 3. 발견된 기술 리스크 & 개선 항목 (우선순위)

- **P1-a. 하객 "재로딩 2회" 버그** — `live.html`: localStorage 캐시로 먼저 렌더하면(`rendered=true`) 이후 fetch로 새 vimeoId가 와도 **그 방문에선 영상을 붙이지 않음**. 당일 입력 시 하객이 두 번 새로고침해야 보임. → fetch 후 `영상 미부착 && vimeoId 있음`이면 즉시 `attachVimeoPlayer` 호출(3줄). 사전 등록으로 가도 안전망으로 필수.
- **P1-b. 갤러리 쇼케이스 8종 Enter 404** — `i/invitations/invitation-0X.html`의 `live.html?e=…` 상대경로가 `/i/invitations/live.html`(없음)로 감. → `/live.html`로 절대경로화(8파일). 실계약 고객용 `i/cover-0X.html`은 이미 정상.
- **P2-a. placeholder 자동 시작 안 됨** — "예식 시각에 시작됩니다"라고 약속하지만 폴링이 없어 하객이 직접 새로고침해야 함. → placeholder 상태면 90초 간격 재조회(fresh 아님·일반 fetch)로 자동 전환. 사전 등록 SOP를 지키면 발동할 일 없지만 약속 이행용 안전망.
- **P2-b. form-to-couple 캐시 무효화 키 불일치** — 폼은 `couple_<id>`를 지우는데 웹훅 v3.4는 `couple_<id>_<view>`(live/online/family/def)로 저장 → **재제출해도 최대 10분 옛값**. → 4개 키 모두 remove(1줄→4줄).
- **P3-a. 예식 전 placeholder 유지 게이트(선택)** — 사전 등록 시 D-3부터 Vimeo 대기 화면이 보이는 게 싫으면, weddingDate·Time 기준 "예식 90분 전"까지는 vimeoId 있어도 placeholder 유지.
- **P3-b. 관리자 페이지 영상등록 카드(선택)** — admin.html에 eventId·vimeoId·hash 입력 UI + 웹훅 프로젝트에 setVimeo 액션(같은 프로젝트라 캐시 즉시 무효화 가능). 시트 직접 입력이 익숙하면 불필요 → 보류 권장.

적용 시 반영 위치: P1-a·P2-a·P3-a = `live.html`(Vercel 자동배포) / P1-b = 갤러리 8파일(〃) / P2-b = `form-to-couple.gs`(**별도 GAS 프로젝트** 재배포 — R3n9Mr 아님) / P3-b = admin.html + `guest-letter-webhook.gs`.

## 4. 권장안 한 줄 요약

**Vimeo 유지 + 라이브 이벤트 D-3 사전 등록 SOP + P1 코드 2건(재로딩 버그·갤러리 404) 즉시 적용, P2 2건 함께 적용 권장.** 당일 운영자는 "송출 시작" 딱 하나만 하면 되고, 하객은 링크만 열면 대기→라이브→다시보기가 전부 자동.

## 5. 남은 결정 (사용자)

1. 송출 방식 기본값: A 라이브 / B 준라이브 (권장 A)
2. Vimeo 요금제: 라이브 지원 플랜 연간 vs 라이브 있는 달만 월결제 (건수 전망에 따라)
3. P3-a placeholder 게이트 넣을지 (사전 등록 시 Vimeo 대기화면 노출 허용 여부)
4. 다시보기 유지 기간·원본 납품 정책 (약관 문구화)
