## 0. 안전 원칙 (전 작업 공통 · 불변)
- **플래그 OFF 기본** — `PAY_CARD_ENABLED!=='true'`면 결제 UI·SDK·웹훅 모두 비활성. 켜기 전 라이브 영향 0.
- **계좌이체(현행) 무영향** — 기존 입금신호→관리자확인 흐름·금액·단계 로직 일절 변경 금지. 결제 "수단"만 추가.
- **금액은 서버가 재계산** — 클라이언트가 보낸 금액 신뢰 금지. `_journeyAmounts` 단일 출처와 일치해야만 기록.
- **멱등** — 같은 단계 중복 확인 금지(이미 '확인'이면 차단). LockService로 직렬화.
- **기록은 기존 함수 재사용** — `adminConfirmPayment/Mid/Balance`를 호출해 기록(컬럼·단계전이·고객알림 일관). 결제수단이 카드든 가상계좌든 동일.

## 3. 작업 항목별 상세

### A. 가상계좌 발급 + 입금 웹훅
**A-1. 발급(프론트→서버 기록)**
- 프론트 `meCardPay.pay(milestone, amount, '가상계좌')` → 토스 SDK `requestPayment('가상계좌', {...})`.
- 토스가 VA 발급(상태 `WAITING_FOR_DEPOSIT`) → successUrl 복귀.
- 복귀 시 `cardPayConfig`가 아니라 **`vaIssued`** 호출: `{token, milestone, paymentKey, orderId, amount, bank, accountNumber, dueDate}` → 서버가 금액검증 후 `가상계좌` 시트에 '발급' 적재 → 화면에 "입금 대기" 카드(계좌번호·기한) 표시.

**A-2. 입금 웹훅(서버, 자동확인)**
- 신규 공개 엔드포인트 `doPost action='vaWebhook'`(또는 별도 webhook URL).
- 수신 시 **payload를 신뢰하지 않고**, `paymentKey`로 **토스 GET `/v1/payments/{paymentKey}` 재조회**(시크릿키) → `status==='DONE'` && `totalAmount===기대금액` 확인.
- `가상계좌` 시트에서 orderId로 행 조회 → 개인코드·단계 확보 → **이미 '확인'이면 멱등 종료** → 아니면 `adminConfirmX(code)` 호출로 기록 + 상태 '입금완료'.
- 웹훅 보안: ① 토스 재조회로 진위 확인(필수) ② 선택 `WEBHOOK_SECRET` 경로 토큰. LockService.
- **만료 처리**: 일1회 트리거가 `가상계좌` '발급' 중 만료 지난 행 → '만료'로 정리(미입금).

**검증**: 발급→가짜입금(샌드박스)→웹훅→해당 단계 '확인' + 마이페이지 반영 + 중복 웹훅 멱등.

### B. 결제수단 차등 노출 (`meCardPay`)
- 단계별 노출 정책(서버 `cardPayConfig`가 단계별 `methods` 배열 반환):
  - 계약금(소액): `['카드','간편결제']` 우선(간편결제 상단) · 가상계좌·계좌이체 보조
  - 중도금·잔금(고액): `['가상계좌']` 기본 · 계좌이체(0%) 보조 · 카드는 "할부 원하면" 접힘 안에
- 프론트는 정책대로 버튼 구성(없는 수단은 미표시).
- **검증**: 단계별로 맞는 버튼만 노출 · OFF면 전무.

### C. 디자인 마감
- `── 또는 ──` 헤어라인 디바이더(기존 톤).
- **현금영수증 입력**: 계좌이체·가상계좌 경로에만 노출, 카드 경로엔 숨김.
- 성공 시 카드 `collapsedCard`로 ✓ 접힘("결제 완료 · 카드/가상계좌").
- **검증**: render-check 구문 통과 + OFF 시 DOM 변화 0.

### D. 묶음결제(중도금+잔금 합계)
- `handleCardConfirm`/`vaWebhook`에 **복수 단계 동시확인** 옵션: `milestones:['중도금','잔금']` → 합계 검증 → `adminConfirmMidBalance(code)` 재사용.
- **검증**: 합계 금액 일치 시 두 단계 동시 '확인'.

### E. (선택) 카드 환불 API
- `adminCancelCard(code, milestone, reason)` → 토스 `/v1/payments/{paymentKey}/cancel` → 상태 되돌림(관리자 전용·adminCall). 노쇼·청약철회 대비.
- **검증**: 샌드박스 부분취소 → 상태 복원.

### F. (선택) 링크페이
- 분납 알림(95_notify)에 결제링크 첨부 발송. 도입 여부 미정(§7 질문).

## ✅ 적용 완료 (2026-07-01 · 코어 분리 실제 반영 · 카드 OFF 유지 · 라이브 무영향)
> "검토했으면 적용" 지시로 SYNC-1·2·3의 코어 수정을 실제 코드에 반영. 관리자 동작은 100% 보존, 카드 경로는 플래그 OFF라 라이브 영향 0. 전 테스트(423건) 통과·문법 확인.

- **SYNC-1·2 해결 — 코어 함수 분리**(`admin.gs`):
  - `_confirmDepositCore(code, {bundle, via})` 신설 — `adminConfirmPayment`에서 가드·번들 뺀 순수 기록 본체.
  - `adminConfirmPayment(code)` = `_requireAdmin()` → `_confirmDepositCore(code, {bundle:true})` (기존 통장 임박 일괄수납 그대로).
  - 카드 계약금(`98_pay_card.gs`) = `_confirmDepositCore(code, {bundle:false, via:'카드'})` — **가드 우회 + 번들 OFF**(계약금 실결제 금액만 확인, 미결제 중도금·잔금이 딸려 확인되던 문제 차단). 임박 일괄은 프론트 '일괄결제' 옵션으로 별도(추가①).
  - 중도금·잔금 카드 = `adminConfirmMid`/`adminConfirmBalance` 그대로 재사용(가드 無·`STAGE_EXCEPTIONS` 차단·안심알림 카드에 그대로 적합).
- **SYNC-3 해결 — 카드=매출전표 → 현금영수증 큐 제외**:
  - `_cashReceiptLedger`(`70_journey.gs`)가 `동의기록.결제수단[키]==='카드'`면 `due=false`+`card:true`로 발급 대상서 제외.
  - 카드 확인 성공 후 `_payMarkCard(code, 원장키)`가 마커 세팅(계약금→'예약금'·중도금→'중도금'·잔금→'잔금'). 동의기록 최신 재읽기로 영수증기준일 등 병존.
  - `결제수단` 마커는 카드 경로에서만 세팅 → **계좌이체·기존 고객은 절대 미세팅 = 원장 동작 완전 동일(무영향)**.
- **함수명 메모**: 최종 코어명은 계약금 전용이라 `_recordPaid`(범용) 대신 `_confirmDepositCore`(계약금 기록 본체)로 확정. 중도금·잔금은 이미 가드 없는 admin 함수라 별도 코어 불필요.
- **아직 안 함(테스트키 생기면 E2E와 함께)**: 가상계좌+웹훅 · 차등노출 · 일괄결제 옵션 · 환불(부분/위약금) · 링크페이 · mypage `card:true` 배지 프론트 표시. 프론트 `card` 필드는 이미 백엔드에서 내려가나 화면 라벨링은 미구현.
