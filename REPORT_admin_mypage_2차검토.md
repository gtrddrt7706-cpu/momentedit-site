## A. 마이페이지 (고객 관점) — 남은 항목

> 디자인(그림자·반경·대비·완료화면·터치)은 이미 반영됨. 아래는 *기능·신뢰* 위주.

- **A1 · 핀치 줌 차단 [P0]** — `viewport`에 `maximum-scale=1.0, user-scalable=no`. 부모님 세대가 계약·청첩장 화면을 못 키워요(WCAG 1.4.4). → 두 속성 제거(입력 자동확대는 input 16px로 이미 방지).
- **A2 · 개인코드 미노출 [P0]** — 코드카드 CSS·복사 핸들러는 있는데 `mypageView` 본문에 `id="mp_code"` 마크업이 없음. 메일 링크로 들어온 고객은 자기 코드를 모를 수 있어 재로그인 곤란. → 환영부/풋터에 코드+복사 재노출(의도적 숨김인지 확인).
- **A3 · 네트워크 회복력 [P1]** — `loadMyState` catch가 항상 로그인으로 튕김. 캐시가 있으면 캐시 화면 유지 + "연결 불안정" 토스트로.
- **A4 · 길잃은 방문자 [P1]** — 로그인 화면에 "아직 신청 전? → 상담 신청" 링크 추가.
- **A5 · 연락 폴백 [P1]** — 카톡 미설정 시 이메일 문의라도 항상 노출.
- **A6 · 결과물 ②③ '곧' [P2]** — "번호 선택·추가 결제 곧 열려요"가 막연 → 예상 시점/안내 한 줄.
- **A7 · NOW 헤드라인 길이 가드 [P2]** — 백엔드 문구에 마침표 없으면 통문장이 헤드라인으로(길어짐).

## B. 관리자 (운영자 관점) — 신규 검토

> admin은 전반적으로 **잘 설계돼 있음**(처리할 일 큐·확인 모달·위험 액션 danger·이력·거울 미리보기·강제변경+사유). 아래는 *진짜 빈 곳*.

### B1 · 제작 카드가 다이닝·식순을 "준비 중"으로 표시 (스테일) — **P0**
- **현상:** `cardProduction`(admin.html:673)이 `🍽 다이닝 · 📜 식순 — 준비 중 (곧)`으로 **하드코딩**. 그런데 우리는 방금 **다이닝·식순 위저드를 출시**했어요.
- **운영자 영향:** 고객이 입력한 **다이닝(인원·식사형태·파트너사·위탁동의)·식순(사회자·식순 12종·축가·30분 타임라인)** 을 **운영자가 전혀 못 봐요.** 예식 준비의 절반이 깜깜이.
- **제안:**
  - `admin.gs adminDetail` → mirror에 `dining`/`ritual` draft 포함(데이터는 이미 `제작임시저장`에 있음).
  - `admin.html cardProduction` → 청첩장처럼 **다이닝·식순 상태(시작전/진행중/완료) + 내용 거울**(인원·메뉴·사회자·순서·타임라인) 렌더.
- **적용:** admin.html(프런트·자동) + admin.gs(재배포).

### B2 · 잔금 관리 전무 — **P0** (잔금 PLAN과 연결)
- admin에 `잔금` 단어 0건. 운영자가 *누가 잔금 미납인지·입금했는지·확인* 할 방법이 없음.
- → `PLAN_잔금_여정_설계.md`의 관리자 부분(제작중/예식완료 행 **잔금 배지** + **[잔금 입금확인]** + 큐 '잔금확인')으로 해결.

### B3 · 핀치 줌 차단 [P1]
- admin.html도 `user-scalable=no`. 운영자가 폰으로 상세(작은 글씨·링크) 볼 때 불편. → 제거.

### B4 · 큐·검색 행에 예식 D-day 없음 [P1]
- D-day는 **파이프라인**에만(`ddayLabel`, 468줄). **처리할 일 큐·검색 결과**엔 없음.
- 운영자 영향: "예식 임박인데 계약/입금 안 된 건"의 급함이 큐에서 안 보임. → 큐·검색 행에도 D-day 배지(임박 건 빨강).

### B5 · 입금 카드에 금액 분해 없음 [P1]
- `cardPayment`는 **총액**만 표시. 통장 대조 시 운영자는 *"이번에 들어올 계약금 = 총액20% − 예약금"* 을 머리로 계산해야 함.
- → 입금 카드에 **계약금/납부액(예약금 차감)/잔금**을 분해 표시(고객 화면과 동일 산식 `_journeyAmounts`).

### B6 · 계약 총액 프리셋 하드코딩 [P2]
- `openSendContract`에 `280만/210만/60만` 하드코딩. 가격 정책 바뀌면 코드 수정 필요. → CONFIG/상수로 분리.

### B7 · 자동 새로고침·갱신시각 [P2]
- 수동 새로고침만. 바쁜 날 신규신청을 놓칠 수 있음. → "n분 전 갱신" 표기 + (선택) 가벼운 폴링/알림.

### 그 외(운영자 디테일)
- **확인 모달**: 모든 액션에 결과 명시·busy·danger 처리 — **훌륭**(유지).
- **되돌리기**: 위험 액션은 확인+이력+강제변경으로 복구 가능 — OK.
- **대비**: 작은 회색 글씨(`--label-soft`)는 마이페이지와 같은 톤 → 운영자 폰 가독성 위해 한 톤 진하게(동일 처리).

## C. 적용 권장 순서

**1) 즉시(프런트만 · Vercel 자동 · 저위험):**
- A1·B3 핀치줌(둘 다) / A3 네트워크 회복력 / A4 상담신청 링크 / A5 연락 폴백 / B4 큐 D-day / B5 금액 분해 / 대비
- → "C1 해줘" 하면 한 번에 반영.

**2) 확인/결정 후:**
- A2 개인코드 노출(숨긴 의도 확인) / B6 가격 config화 / B7 자동 새로고침 범위

**3) 백엔드 재배포 필요:**
- **B1 제작 3트랙 거울**(admin.gs mirror에 dining/ritual 추가 + admin.html 렌더) — *다이닝/식순 출시했으니 가장 시급*
- **B2 잔금 전체**(잔금 PLAN)

### 마무리
마이페이지는 디자인까지 다듬여 **고객용으로 거의 완성**, 남은 건 접근성·신뢰 5개(A1~A5). 관리자는 구조가 탄탄한데 **딱 두 곳이 비어 있어요 — ①다이닝·식순(출시했는데 운영자 화면이 옛날) ②잔금.** 이 둘이 운영자 체감 1·2순위예요.
**"C1 해줘"** 또는 번호로 골라주시면 그대로 반영하고, B1·B2는 잔금 PLAN과 함께 백엔드까지 묶어 구현할게요.
