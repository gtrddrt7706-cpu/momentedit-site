# 모먼트에디트 · 결제 시스템 기획안 (마이페이지 통합)

> 작성 2026-06-30 · 목적: 카드·간편결제를 **기존 마이페이지 결제 카드 틀에 자연스럽게 녹여** 노출하는 시스템·디자인·UX 설계.
> 전제: 백엔드(`98_pay_card.gs`)·프론트(`mypage.html` `meCardPay`)는 사전 구축 완료(플래그 OFF). 이 문서는 "켤 때 어떻게 보이고 동작할지"의 설계도.
> (수수료·결제수단 선호 등 시장 근거는 `deep-research` 결과로 보강 예정)

---

## 0. 핵심 원칙 (먼저 합의할 방향)

1. **계좌이체 우선, 카드는 "편의 옵션"** — 계좌이체는 수수료 0이자 브랜드(추가금 없는 단일가)와 결이 같다. 카드·간편결제는 건당 2~3% 비용이므로, **시각적 1순위는 계좌이체**로 두고 카드는 차분한 2순위로 병행한다. 강요·상단 강조 X.
2. **기존 틀을 깨지 않는다** — 새 페이지·새 카드 만들지 않고, **지금의 결제 카드(계약금·중도금·잔금) 안에** 결제수단 한 줄만 더한다. 디자인 토큰(`consult-card`·`cc-btn`·`cc-eyebrow`·`cc-note`)을 그대로 쓴다.
3. **단계는 그대로** — 계약금 10% → 중도금 40% → 잔금 50% 마일스톤·금액·기한 로직은 손대지 않는다. 결제 "수단"만 추가.
4. **신뢰·차분함** — 고액(280만원) 결제라 "안전하게 처리됨"이 느껴지게. 압박·할인팝업 없음.

---

## 1. 결제수단 구성 (무엇을 노출하나)

토스페이먼츠 **하나**로 아래를 한 결제창에 통합(별도 가맹 불필요):

| 수단 | 성격 | 우리 비용 | 노출 |
|---|---|---|---|
| **계좌이체(무통장)** | 현행 · 입금신호→확인 | **0%** | **1순위(기본)** |
| 카드 | 즉시 확인 · 할부 가능 | ~2~3% | 2순위(편의) |
| 카카오페이·네이버페이 | 간편·친숙 | ~2~3% | 카드 결제창 내 선택지로 포함 |
| 가상계좌(선택) | 무통장 자동대사 | 건당 수백 원 | 추후 검토 |

> 카카오페이/네이버페이는 "따로 붙이는 것"이 아니라 **토스 결제창 안의 결제수단**으로 함께 뜬다 → 추가 개발·가맹 거의 없음.

## 2. 노출 위치 · 계층 (기존 카드 어디에)

현재 결제 카드 구조(예: 잔금):
```
[consult-card]
  Balance (cc-eyebrow)
  안내문(금액·디데이)
  계좌 안내 + [계좌 복사]
  입금자명 입력
  현금영수증 입력(선택)
  [cc-actions]  [입금했어요]  ← cc-btn-seal(브랜드 브라운·1순위)
  cc-note: 입금 확인 후 반영
```

**여기에 딱 한 블록만 추가** — 계좌이체 영역 아래, 얇은 구분선과 함께:
```
  ── 또는 ──
  [ 카드·간편결제로 결제 (₩2,800,000) ]   ← cc-btn(아웃라인·2순위) = me-card-pay
  cc-note(작게): 카드·카카오페이·네이버페이 · 즉시 확인돼요
```

계층 의도:
- **입금했어요** = `cc-btn-seal`(채움·브라운) → 시선 1순위(계좌이체 = 무수수료).
- **카드로 결제** = `cc-btn`(아웃라인) → 차분한 2순위(편의). 색·크기로 "보조" 신호.
- 둘 사이 `── 또는 ──` 얇은 디바이더로 "두 길" 명확히.

적용 카드: **계약금·중도금·잔금 3곳 동일 패턴**(이미 `mp_paySignal`·`mp_mSignal`·`mp_bSignal` 후크에 `meCardPay`가 버튼 주입).

## 3. 디자인 디테일 (기존 토큰으로)

- **버튼**: `class="cc-btn me-card-pay"` (아웃라인). 텍스트 `카드·간편결제 (금액)`. 폭은 입금했어요와 동일(풀폭), `margin-top:8px`.
- **아이콘(선택)**: 버튼 좌측에 작은 카드/카카오 아이콘은 **넣지 않음**(미니멀 유지) 또는 단색 1개만. 브랜드는 절제가 강점.
- **색**: 브랜드 브라운(`--seal` #6B2A24)·아이보리·골드(#c9a866) 팔레트 유지. 카드 버튼은 브라운 외곽선 + 투명 배경.
- **디바이더**: `── 또는 ──` = 가는 헤어라인 + 가운데 작은 회색 글자(기존 `past-toggle`/`cc-note` 톤).
- **완료 상태**: 카드 결제 성공 → 해당 카드는 다른 완료 카드처럼 **✓ 한 줄로 접힘**(`collapsedCard` 패턴 재사용) → "결제 완료 · 카드".
- **현금영수증 입력**: **계좌이체일 때만 노출**. 카드 결제는 카드매출전표(소득공제 자동)라 현금영수증 칸은 카드 경로에선 숨김 → 혼동 제거. (중요 디테일)

## 4. UX 흐름 (단계별 · 상태별)

### 정상 흐름
1. 결제 카드에서 **[카드·간편결제]** 탭
2. 버튼 → `결제창 여는 중…`(disabled) → 토스 SDK 결제창(카드/카카오페이/네이버페이 선택)
3. 결제 완료 → 마이페이지로 복귀(`?me_pay=1`)
4. 자동으로 서버 승인(`cardConfirm`) → "결제가 완료되었어요" → 카드 ✓ 접힘 + 다음 단계로

### 실패·취소
- 취소(`me_pay=0`): "결제가 취소되었어요" 안내 후 카드 그대로(재시도 가능).
- 승인 실패: "결제 확인에 실패했어요. 디렉터에게 문의해 주세요." + 계좌이체 옵션 그대로 보임(대체 경로 보장).

### 분할·묶음
- **계약금/중도금/잔금 각각** 카드 결제 가능.
- **묶음결제(중도금+잔금 함께 · 임박 계약 일괄)**: 카드도 **합계 1회 결제** 지원하도록 확장 필요(현재 백엔드는 마일스톤 단건 → 묶음 시 합계 금액·복수 마일스톤 동시 확인 로직 추가). → 적용 로드맵 2단계.

### 로딩·피드백
- 결제창 호출 지연 대비 버튼 로딩 텍스트.
- 복귀 후 승인 처리 중 짧은 스피너/문구(기존 `mpAlert`·`loadMyState` 재사용).

## 5. 카피(문구) — 브랜드 톤 (합니다체·가운뎃점)

- 디바이더: `또는`
- 카드 버튼: `카드·간편결제 (₩2,800,000)`
- 보조 설명(cc-note): `카드·카카오페이·네이버페이로 결제하실 수 있어요. 즉시 확인돼요.`
- 성공: `결제가 완료되었어요. 감사합니다.`
- 취소: `결제가 취소되었어요. 다시 시도하실 수 있어요.`
- 실패: `결제 확인에 실패했어요. 디렉터에게 문의해 주시면 바로 도와드릴게요.`
- (계좌이체 우선 암시는 카피로 강요하지 않음 — 배치·색으로만 표현)

## 6. 모바일 고려

- 결제창은 토스가 모바일 최적(앱 전환 포함) → 우리는 버튼만 풀폭·터치 영역 충분히.
- 복귀(리다이렉트) 흐름이라 **세션 유지**가 핵심: `sessionStorage`에 단계 저장(이미 구현) · 복귀 시 토큰으로 재인증(`getToken`).
- 카카오페이/네이버페이는 앱 전환 후 복귀 → `successUrl` 동일 페이지 처리(구현됨).

---

## 7. 마진 vs 편의 (전략)

- **계좌이체(0%)를 항상 1순위로 유지** → 대다수가 계좌이체 선택하도록 자연 유도(비용 절감).
- 카드/간편결제는 "원하는 분만" → 편의·전환율 보완. 고액이라 카드 할부를 원하는 고객 흡수.
- (선택) **카드 결제 시 무이자 할부 행사**는 PG·카드사 프로모션 활용(우리 부담 최소). → 연구결과 확인 후 결정.
- 수수료 절감: 영세·중소 우대수수료 적용 여부 확인 · 가상계좌(건당 정액)로 고액 일부 흡수 검토.

## 8. 시스템 매핑 (이미 만든 것과 연결)

| 기획 요소 | 구현 위치(이미 존재) |
|---|---|
| 카드 버튼 주입 | `mypage.html` `meCardPay.attachOne`(`mp_paySignal`/`mp_mSignal`/`mp_bSignal`) |
| 결제창 호출 | `meCardPay.pay` (토스 SDK 지연로드) |
| 복귀 승인 | `meCardPay.handleReturn` → `cardConfirm` |
| 금액 검증·승인·기록 | `98_pay_card.gs` `handleCardConfirm`(서버 금액 재계산·토스 confirm·기존 관리자확인 재사용) |
| 활성 토글 | ScriptProperty `PAY_CARD_ENABLED`(OFF) |

→ **기획의 대부분이 이미 코드로 준비됨.** 남은 건 (a) 디자인 마감(디바이더·현금영수증 분기·완료 접힘), (b) 묶음결제 카드 확장, (c) 샌드박스 E2E.

## 9. 적용 로드맵

- **1단계(지금·완료)**: 백엔드 + 프론트 단건 결제 구축(OFF).
- **2단계**: 디자인 마감(`또는` 디바이더 · 카드 경로 현금영수증 숨김 · 성공 시 ✓ 접힘) + 묶음결제(중도금+잔금 합계) 카드 확장.
- **3단계**: 토스 테스트키 → 샌드박스 E2E(결제·실패·취소·환불) 검증 → 검증완료 OFF.
- **오픈 후**: 통신판매업 신고 → 토스 라이브 계약 → 라이브 키 교체 + `PAY_CARD_ENABLED=true`.

## 10. 리스크 · 엣지

- **금액 위변조**: 서버 재계산값과 불일치 시 기록 거부(구현됨).
- **이중결제/이중확인**: 멱등(이미 확인이면 차단) + LockService(구현됨).
- **승인 후 기록 실패**: 토스는 성공인데 시트 기록 실패 → 로그 `기록경고` 남김 → 관리자 수동 확인 필요(개선 여지: 재시도/알림).
- **카드 한도 초과(고액)**: 280만원 1회 결제 한도 막힐 수 있음 → 안내 문구 + 계좌이체 대체 경로 항상 노출로 흡수.
- **환불**: 카드 환불은 토스 환불 API 필요(현재 미구현) → 취소·환불 흐름은 별도 설계(로드맵에 추가).

---

## 11. 결론(권장)

- **"계좌이체 1순위 + 카드/간편결제 2순위"를 기존 결제 카드 안에 한 블록으로** 녹이는 것이 최적. 새 화면 없이 자연스럽고, 마진(계좌이체)·편의(카드)를 동시에 잡는다.
- 디자인은 **절제(아웃라인 버튼·얇은 디바이더·완료 접힘)** 로 브랜드 톤 유지.
- 코드는 이미 80% 준비 → 디자인 마감 + 묶음/환불 보강 + E2E만 남음.
