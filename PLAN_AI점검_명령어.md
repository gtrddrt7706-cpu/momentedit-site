## 운영 수칙 (모든 점검 공통 · 비용 가드)

- **비용 기준**: 풀배터리(78종 + G0 가드) 1회 ≈ 20분 · 약 3~4천원. 부분 실행(ONLY)은 수백 원.
- **라운드 상한**: 풀배터리는 ①베이스라인 1회 ②(수정이 있었으면) 최종 확인 1회, **총 2회까지만**. 중간 재검은 전부 부분 실행.
- **부분 실행**: `scripts/.sim-run`에 `ONLY=이름정규식`을 적어 트리거하면 일치 시나리오만 돈다. 예: `round7 ONLY=N5|E11` → 수정한 케이스+인접 케이스만 재검.
- **CHECK ≠ 실수**: 채점은 키워드 매칭이라 AI의 표현 다양성 때문에 풀런 만점은 확률 게임(라운드마다 다른 1~3개가 걸리는 게 정상). 목표는 "CHECK 0"이 아니라 **"원문 판독 후 진짜 실수 0"**. 오탐만 남으면 종료한다.
- **CHECK 처리**: ①채점 오탐 → 채점만 보정하고 **재실행하지 않는다**(다음 풀런에서 자연 검증) ②진짜 실수 → 최소 diff로 수정 → main 배포 → **ONLY 부분 재검**(해당+인접 케이스).
- **회귀 주의(실사례)**: "공휴일=280" 규칙을 추가하자 모델이 공휴일 여부에만 집중해 "토요일=평일가" 회귀가 났음. 지식의 분기 규칙은 반드시 **①②③ 순서형**으로 쓰고("①토·일 280 ②공휴일 280 ③둘 다 아니면 210"), 수정 후 인접 케이스를 같이 재검할 것.
- **채점 작성법**: expect는 의미 계열 OR로 넓게(본질 강점이면 5종 전부 + `\[ESC\]` 에스컬레이션 신호 인정), forbid는 **단정 패턴만** 좁게(부정·대조 문맥 "평일이면 210인데 공휴일은 280"이 걸리지 않게). 시간대 3개를 "어느 시간대가 편하세요?" 메뉴로 나열하는 건 가용성 누수가 아님(니즈 질문의 표준 형태 · 채점기 menuAsk 예외 처리됨).
- **동시 실행 금지**: 트리거 전 Actions에 시뮬 in_progress가 없는지 확인. `.sim-run` 변경은 **작업 브랜치에만** 푸시(main 동시 푸시 = 이중 실행 = 2배 과금 + 동시 부하로 500 오류). 일시 5xx는 하니스가 1회 재시도로 흡수함.
- **리베이스 재생 트리거 금지(2026-06-13 실사례)**: `.sim-run`을 건드린 커밋이 브랜치에 있는 채로 rebase + force-push 하면, 재생된(새 SHA) 트리거 커밋이 푸시에 포함돼 **시뮬이 또 돈다**(풀런 2개 중복 실행으로 약 3~4천원 추가 과금 · 진행 중 Actions 취소는 통합 토큰 권한 403으로 불가). 규칙: ①시뮬을 한 번이라도 트리거한 브랜치는 리베이스하지 않는다(main 동기화가 필요하면 merge) ②수정 커밋의 main 반영은 **체리픽**으로(트리거 커밋이 main에 딸려가지 않게): `git worktree add /tmp/mainfix origin/main` → 거기서 `cherry-pick <수정sha>` → `push origin HEAD:main` → worktree 제거.
- **오프라인 먼저**: 단계 경계·가중치·날짜 계산 등 결정적 로직은 라이브 말고 `node scripts/test-schedule-advisor.js`(무료)로 검증. 라이브는 "문장 톤·환각·표현"만.
- **돈 수치는 PASS여도 사람 눈으로**: 채점기는 키워드만 봐서 "60~30일 20%" 같은 경계 하루 오차를 못 잡는다(실사례). 위약금·수수료 등 표 데이터가 답변에 나오면 구간·비율을 원문 표와 대조하고, KB의 표에는 "그대로 인용·경계 합치기 금지" 지침을 함께 적는다.
- **날짜·요일 짝도 사람 눈으로(2026-06-13 실사례)**: "11월 7일 토요일"처럼 날짜와 요일이 어긋난 표기는 채점기가 못 잡는다(2027-11-07은 실제 일요일 · 고객이 요일을 오인한 채 그 날짜를 픽스할 위험). 서버에 결정적 가드를 넣어뒀지만(`snapWeekday` 추출 스냅 · `fixWeekdayText` 표기 교정 · 오프라인 회귀 J1~J9), 스케줄 답변의 날짜·요일 표기는 판독 때 달력과 대조한다.
- **handoff 한계**: brief(분류·요약·제안답변·확신도)는 고객 응답에 포함되지 않음 → 라이브에선 `ok`·`delivered=true`만 검증 가능. 품질은 관리자 페이지 카드에서 사람이 확인.
- **fail-closed 원칙(2026-06-12 더블부킹 교훈)**: 가용성(점유) 조회가 실패하면 **빈 맵=전부 가능으로 가장하지 않는다**(가능·마감 단정 금지·안전 안내). 점유 라우팅 누락은 채점 키워드로 못 잡으니, 시뮬은 맨 앞에서 무료 핑(`G0 가용성 라우팅 생존` · `/api/schedule-advisor {probe:true}` → `availSource:'gas'`)으로 결정적 확인하고, 죽었으면 즉시 빨간 실패로 끊는다. 같은 이유로 `delivered`는 `r.ok`만이 아니라 응답 `ok·id`까지 검증한다.
